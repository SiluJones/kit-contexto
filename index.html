Início

Contexto que viaja entre conversas.

A regra que mantém tudo leve. O Projeto injeta os arquivos em toda mensagem. Se eles crescem sem limite, você recria o problema de token que queria evitar. Por isso o sistema separa contexto (leve, recarregado sempre) de histórico (no Git, lido só sob demanda). O peso do Projeto fica praticamente constante.

↻ Contexto — fica no Projeto

⌗ Histórico — fica no Git

Lido só quando você pede, então pode ser detalhado. A pasta logs/ inteira e o que sai do STATUS quando resolvido. Nunca recarregados a cada conversa.

Os arquivos deste nicho

Resposta direta sobre o peso: bem gerenciados, esses arquivos não viram peso problemático. O segredo é o STATUS rolante (item resolvido sai dele) e os logs fora do contexto carregado. Assim o tamanho do Projeto fica quase constante por mais que ele cresça.

→ Por onde começar

1. Abra Instruções e monte o bloco para colar no Projeto do Claude.
2. Pegue o prompt certo em Prompts (C para projeto novo, D para existente, E para transferir uma conversa pesada).
3. Baixe os Templates e suba tudo no Git.

Mecânica & passo a passo

      
Tokens e fluxos.

      
Por que os tokens "acabam", quando trocar de conversa, e o caminho exato para cada situação.


      
Por que os "continue" acontecem

      
A cada mensagem, o modelo recebe toda a conversa desde o início como entrada — não só a última mensagem. Quanto maior o histórico, menos espaço sobra para a resposta. O "continue" repetido é o limite de saída por vez, não o limite total. É um copo: quanto mais histórico enche, menos espaço para despejar resposta nova.

      
Memória (toggle das configurações) adiciona só um pequeno resumo de conversas antigas. O impacto é marginal — o vilão é o histórico crescente da própria conversa atual.

Regra prática: mandou "continue" mais de 2 vezes na sessão? Abra uma nova com o pacote de contexto. Sai mais barato que continuar.


      
Os três fluxos

      
Projeto novo

      
1. Crie o Projeto no Claude.ai
2. Cole as Instruções · suba os templates do nicho como referência
3. Use o Prompt C com suas ideias
4. Claude gera o necessário + os .md preenchidos
5. Substitua os templates pelos preenchidos · suba no Git
6. Próximas conversas → só o Prompt A


      
Projeto existente (ainda sem o sistema)

      
1. Crie o Projeto · cole as Instruções
2. Suba o material atual (e os templates como referência)
3. Use o Prompt D → Claude gera os .md com a realidade atual
4. Substitua os templates pelos preenchidos · suba no Git
5. Próximas conversas → só o Prompt A


      
Conversa pesada — transferir agora

      
1. Na conversa atual → Prompt E (não dá para subir nada nela; o Claude já conhece a estrutura)
2. Crie o Projeto novo · cole as Instruções
3. Suba os .md gerados + LOG-TEMPLATE.md + os arquivos do projeto
4. Conversa nova → Prompt F
5. Próximas conversas → só o Prompt A


      
O LOG-TEMPLATE.md é o único que fica permanente no Projeto sem ser substituído — o Claude o usa de referência para gerar os logs em logs/.


      
Detalhes que importam

      

        • Maiúsculas no prompt não fazem diferença técnica. O que importa é estrutura clara,
        contexto antes da tarefa, exemplos e formato de saída definido.

        • Instruções vs prompt: instruções para comportamento permanente; prompt para o contexto
        do dia. Complementares, não substitutos.

        • Você nunca precisa colar o conteúdo dos .md no prompt. Suba no Projeto uma vez (ou anexe
        pelo clipe) e o Claude lê sozinho.

        • Trocar de nicho no meio do projeto: raramente faz sentido — os arquivos foram pensados
        para o domínio. Se o foco mudou de verdade, melhor criar um Projeto novo.
